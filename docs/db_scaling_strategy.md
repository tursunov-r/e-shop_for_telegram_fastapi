# Стратегия масштабирования БД для проекта SFMShop

## Репликация

### Когда использовать
- Для повышения отказоустойчивости и доступности.
- Для распределения нагрузки на чтение (например, аналитические запросы).
- Для обеспечения резервного копирования и аварийного восстановления.

### Преимущества
- Высокая доступность: при падении основного узла можно переключиться на реплику.
- Масштабирование чтения: запросы SELECT можно направлять на реплики.
- Простота настройки по сравнению с шардированием.

### Недостатки
- Запись всегда идёт на основной узел → узкое место при высоких нагрузках.
- Реплики могут отставать (eventual consistency).
- Увеличение затрат на инфраструктуру.

### Применение в SFMShop
- Репликация полезна для аналитики: отчёты по заказам и пользователям можно выполнять на репликах.
- Основной узел обслуживает транзакции (создание заказов, обновление баланса).
- Реплики обслуживают запросы статистики (например, количество заказов за день).

---

## Шардирование

### Когда использовать
- При очень больших объёмах данных (миллионы пользователей, миллионы заказов).
- Когда нагрузка на запись становится узким местом.
- Когда данные можно логически разделить (например, по регионам или по диапазону user_id).

### Преимущества
- Горизонтальное масштабирование: данные распределяются по нескольким узлам.
- Увеличение производительности записи.
- Возможность хранить огромные объёмы данных.

### Недостатки
- Сложность настройки и администрирования.
- Запросы, затрагивающие несколько шардов, становятся дорогими.
- Сложнее поддерживать транзакционную целостность.

### Применение в SFMShop
- Если проект выйдет на уровень миллионов заказов в день, можно шардировать таблицу `orders` по `user_id` или по регионам.
- Это позволит распределить нагрузку на запись и хранение.
- Отчёты по конкретному пользователю будут выполняться быстро, но глобальная аналитика потребует агрегации с нескольких шардов.

---

## Компромиссы

- **Eventual consistency**: при репликации данные на репликах могут отставать от основного узла. Это приемлемо для аналитики, но не для транзакций.
- **Сложность настройки**: репликация проще, шардирование требует сложной конфигурации и мониторинга.
- **Производительность**: репликация улучшает чтение, но не запись; шардирование улучшает запись, но усложняет глобальные запросы.

---

## Итоговая стратегия для SFMShop

1. На ранних этапах проекта использовать **репликацию**:
   - Основной узел для транзакций.
   - Реплики для аналитики и отчётов.
2. При росте нагрузки и объёмов данных перейти к **шардированию**:
   - Разделить таблицу заказов по пользователям или регионам.
   - Использовать репликацию внутри каждого шарда для отказоустойчивости.
3. Обеспечить баланс между производительностью и целостностью:
   - Критические транзакции всегда выполняются на основном узле.
   - Аналитика допускает eventual consistency.
