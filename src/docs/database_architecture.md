# Гибридная архитектура БД для проекта SFMShop

## PostgreSQL — основные данные
**Назначение:**
- Хранение критически важных транзакционных данных.
- Пользователи, заказы, товары, платежи.

**Почему PostgreSQL:**
- Поддержка ACID-транзакций.
- Богатый SQL-функционал для сложных запросов.
- Подходит для строгой целостности данных.

**Примеры данных:**
- Таблица `users` — данные о пользователях, баланс.
- Таблица `orders` — заказы и их связь с пользователями.
- Таблица `products` — товары, цены, остатки.

---

## Redis — кэш, сессии, очереди
**Назначение:**
- Быстрый доступ к временным данным.
- Хранение пользовательских сессий.
- Очереди для фоновых задач (например, обработка заказов).

**Почему Redis:**
- Очень высокая скорость работы с данными в памяти.
- Поддержка TTL (время жизни ключей).
- Удобен для реализации очередей и pub/sub.

**Примеры данных:**
- Кэш популярных товаров.
- Сессии пользователей.
- Очередь задач на обработку платежей или уведомлений.

---

## MongoDB — логи, аналитика, события
**Назначение:**
- Хранение неструктурированных данных.
- Логи, события, аналитика.

**Почему MongoDB:**
- Гибкая схема: можно хранить разные структуры логов.
- Удобно для аналитики и агрегации.
- Хорошо масштабируется при больших объёмах данных.

**Примеры данных:**
- Коллекция `logs` — ошибки, доступы, системные события.
- Коллекция `analytics` — статистика по заказам и пользователям.
- Коллекция `events` — бизнес-события (например, «пользователь зарегистрировался»).

---

## Поток данных и связи

1. **Пользователь делает заказ**  
   - Данные заказа сохраняются в PostgreSQL (`orders`, `order_items`).  
   - Баланс пользователя обновляется в PostgreSQL (`users.balance`).  
   - Событие заказа отправляется в Redis (очередь для фоновой обработки).  

2. **Фоновая обработка заказа**  
   - Redis очередь передаёт задачу воркеру.  
   - Воркер обновляет статистику в MongoDB (`analytics`).  
   - Лог операции сохраняется в MongoDB (`logs`).  

3. **Кэширование**  
   - Популярные товары и результаты поиска кэшируются в Redis.  
   - При обновлении товара в PostgreSQL кэш сбрасывается.  

---

## Компромиссы

- **PostgreSQL**: строгая целостность, но медленнее при больших объёмах аналитики.  
- **Redis**: высокая скорость, но данные временные (нельзя хранить критические данные).  
- **MongoDB**: гибкость и масштабируемость, но нет строгой транзакционной целостности.  

**Итоговое решение:**
- PostgreSQL для транзакций и критических данных.  
- Redis для кэша, сессий и очередей.  
- MongoDB для логов, аналитики и событий.  

Такой гибридный подход позволяет одновременно обеспечить целостность транзакций, высокую скорость работы и гибкость аналитики.
